## ⚔️ 反幻觉代码法典

### 禁止模式

| 模式 | 症状 | 替代 |
|------|------|------|
| 幽灵调试 | 仅打印/日志 | 真实逻辑或抛错 |
| 假返回值 | 硬编码成功/空值 | 真实数据或抛错 |
| 注释即实现 | TODO/FIXME 无代码 | 骨架或抛错 |
| 永远成功桩 | 验证总成功 | 失败路径或抛错 |
| 空函数体 | 空/占位符 | 抛错或真实逻辑 |
| 幻觉数据 | 硬编码假指标 | 真实计算或外部源 |
| 禁用标记 | @ts-ignore/any | 修复类型错误 |
| 占位标识 | mock/stub/placeholder | 真实逻辑或抛错 |
| 降级标识 | fallback/hack/temp | 正确方案 |

**判断**：放生产会静默失败 = 禁止

### 🚫 绝对禁止标记

`|Null|TODO|FIXME|HACK|XXX|BUG|Not implemented|Coming soon|Placeholder|Mock data|模拟|Fake data|Dummy data|Stub|Fallback|Workaround|Temporary|Quick fix|@ts-ignore|@ts-nocheck|@ts-expect-error|eslint-disable|any`

### 代码扫描

提交前扫描：注释/标识符/字符串/类型/返回值/函数体
违规：停止→报告→修复→重新实现

### 🚫 拒绝名义实现

**名义实现**：形式上实现，实际无效（最危险）

**6 特征**：名义数据库操作(调用不用结果/不检查成功/不处理回滚)、名义验证(调用忽略结果/总通过/不阻止无效输入)、名义错误处理(捕获只记日志/不补救/不检查错误码)、名义业务逻辑(调用不应用结果/计算不用/所有分支同样)、名义集成(调用 API 不处理响应/发消息不确认/不真正重试)、名义状态管理(更新不影响行为/状态转换无效/保存不读)

**识别**（任一否=名义）：返回值被用？副作用被观察？删除调用行为会变？错误被正确处理？输出影响后续？

**检测**：所有查询结果被用、所有验证失败阻止操作、所有异常适当处理(非仅日志)、所有外部调用响应被检查、所有计算结果被用、所有状态变化影响行为、所有返回值被检查、删除任何行都改变行为

**发现**：停止→识别根因→完全重写→验证有效

### 自我纠正

问：*"真实用户调用会做实际意义的事吗？"*
否→不提交→实现真实逻辑 OR 显式抛错

### ⚠️ 零容忍

**拒绝**：用禁止标记、提交空函数体、用类型禁用、硬编码假数据、静默失败
**例外**：测试文件 mock/stub/fake(明确标注)、错误消息"未实现"(运行时抛出)、文档未来计划

## 🛡️ 强制实现完整性

### 核心

禁止"看起来能运行实际无效"
必须：真实数据流、明确错误路径、可验证副作用、明确未完成标记

### 三阶段

**Phase 1 骨架**：定义接口/类/函数签名、未实现抛错注明"Phase 1: 骨架 - 待后续"、禁止假返回/空数据/TODO
**Phase 2 核心**：真实数据处理、验证输入、连真实依赖、真实成功/失败路径、禁止查询后直接成功/不检查响应/只成功分支
**Phase 3 完整**：完整错误处理、日志、性能优化(缓存)、监控指标、完整输入验证、所有异常处理

### 阶段标记

`PHASE_1_SKELETON`(接口定义未实现)、`PHASE_2_PARTIAL`(核心实现缺功能)、`PHASE_3_OPTIMIZATION`(基本完整待优化)
说明：缺什么/何时实现/为何不现在

### 强制 TDD

流程：写测试(正常+异常)→运行(应失败)→实现(通过)→重构(保持通过)
覆盖：成功/失败/边界/异常

## ⏱️ 任务执行控制与恢复

### 理念

任务可控、可中断、可恢复。避免长时间卡住或无限循环

### 超时与卡住检测

**超时阈值**：单函数 5 分钟、单测试 3 分钟、单 bug 10 分钟、单 Phase 30 分钟、单模块 60 分钟
**卡住信号**：连续 3 次同方法失败、同错误重复 3 次、连续 3 轮无物理侧效应（文件操作/命令成功）、思维陷入信息增量停滞、等待外部资源超 2 分钟

### 自动终止与跳过

**立即终止**：死循环/无限递归、连续失败 5 次、超时 3 倍、资源耗尽、不可恢复错误
**自动跳过**：非核心超时 2 倍、优化失败 3 次、测试环境问题、外部依赖不可用有降级
**跳过策略**：

```
IF (超时 OR 卡住):
    保存 checkpoint
    标记 SKIPPED
    记录原因上下文
    继续下一任务
IF (所有完成后有 SKIPPED):
    用不同方法重试
    再失败报告长官
```

### Checkpoint 机制

**保存时机**：完成 Phase、完成模块、任务执行前、检测可能失败前、手动触发

**内容**：

```json
{
  "timestamp": "时间戳",
  "task_id": "任务 ID",
  "phase": "当前阶段",
  "completed_tasks": ["已完成"],
  "current_task": "当前",
  "context": {
    "files_modified": ["修改文件"],
    "dependencies_added": ["新增依赖"],
    "config_changes": ["配置变更"]
  },
  "state": {"variables": "变量", "progress": "进度%"},
  "next_steps": ["下一步"]
}
```

**恢复**：

```
IF (失败 OR 中断):
    加载最近 checkpoint
    恢复状态
    分析失败原因
    选择替代方案
    从 checkpoint 继续
```

### 自动修复策略（5 层级从低到高）

**Level 1 重试**：简单重试(最多 3 次)、清缓存重试、重启服务重试
**Level 2 降级**：简化实现、跳过非核心优化、同步替代异步、本地替代远程
**Level 3 绕过**：标记 TODO(仅无法实现时)、用 mock(仅测试环境)、临时禁用(记录原因)
**Level 4 重新设计**：分析根因、设计替代方案、从 checkpoint 重新开始、用不同技术栈
**Level 5 人工介入**：报告详细错误、提供已尝试方案、建议解决方向、等待决策

### 修复决策树

```
失败
├ 超时？
│ ├ 是→检查卡住
│ │ ├ 死循环→终止→L4 重新设计
│ │ ├ 等外部资源→检查降级
│ │ │ ├ 有→L2 降级
│ │ │ └ 无→L5 人工
│ │ └ 计算复杂→L3 绕过或 L4 重新设计
│ └ 否→继续
├ 重复失败？
│ ├ <3 次→L1 重试
│ ├ 3-5 次→L2 降级或 L3 绕过
│ └ >5 次→L4 重新设计或 L5 人工
├ 核心功能？
│ ├ 是→不跳过→L4 或 L5
│ └ 否→跳过→保存 checkpoint→继续
└ 有替代？
  ├ 是→L4 重新设计
  └ 否→L5 人工
```

### 监控与报告

**监控指标**：执行时间、失败重试次数、checkpoint 次数、跳过数量、修复成功率

**异常报告**：

```
⚠️ 异常
任务: [名]
状态: [超时/卡住/失败]
已尝试: [方案 1][2][3]
原因: [根因]
Checkpoint: [最近 ID]
建议: [修复建议]
决策: [自动修复/跳过/人工]
```

### 恢复与继续

**恢复流程**：检测异常→保存 checkpoint→分析原因→选择修复策略(L1-5)→执行修复→验证成功→成功从 checkpoint 继续/失败升级更高 Level

**继续规则**：修复成功立即继续、跳过继续下一个、人工介入等待指令、所有完成回顾 SKIPPED
